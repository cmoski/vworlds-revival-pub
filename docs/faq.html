Who was the Microsoft Virtual Worlds Group?
A team within Microsoft Research, researching and developing tools for communication, collaboration, and community on the internet. They built one of the first collaborative 3D virtual world platforms — years before Second Life, a decade before VRChat. The group released the platform source code under a research license in February 2000, and the original site at vworlds.org ran until roughly 2005 before going offline. Their research site was at research.microsoft.com/vwg/.
What was vworlds.org?
vworlds.org was the community hub for developers and users of the Virtual Worlds platform. It hosted the source code browser, download links, mailing list archives, documentation, a sample gallery, and bug tracker. The site is long gone — all that remains are Wayback Machine snapshots. This GitHub Pages site is a reproduction of the original in style and spirit, updated for the revival project.
What is the Virtual Worlds platform?
The platform enables users to build persistent online multi-user applications with a flexible UI. It is based on a distributed client-server architecture. The persistence and replication of property changes is handled automatically by the platform to free the developer from having to deal with lower level details. The platform is composed primarily of COM objects, which are built into a few core DLLs, as well as some ActiveX Controls and the server executable.

At the UI level, the original design focused on web integration to provide developers the freedom to integrate other web technologies within an Internet Explorer based UI. This allows for text, 2D or 3D UI, or a combination, for your clients.
What are the platform requirements?
Original (2000): Windows 98 2nd Edition or Windows 2000, Internet Explorer 5.

Revival (2026): Windows 10 or 11 (64-bit — runs as 32-bit via WoW64), Visual C++ 2022 Redistributable (x86), Administrator privileges for initial COM registration. Hardware-accelerated graphics recommended. No external DirectX SDK needed — the legacy D3DRM runtime is bundled.
What is VWorlds Revival?
A preservation effort to modernize the original VWorlds v1.5 source code to compile and run on Windows 11 with Visual Studio 2022. The original code was built for Visual C++ 6.0, Windows 98, and APIs that no longer exist in modern SDKs. Hundreds of modernization fixes were required to get everything compiling and running again — see the Changelog for details.
For what version is the source code?
The revival is based on the v1.5 source release from February 2000. This was the final public release, following v1.0 from November 1998. Major improvements over v1 included server performance, security, additional multimedia/3D graphics features, and a more stable platform.
How do I build from source?
You need Visual Studio 2022 with MFC/ATL for v143 build tools and Windows SDK 10.0. See the Building from Source page for complete instructions. Or just download a pre-built release — no build tools required.
Do I need to install DirectX?
No. The legacy Direct3D Retained Mode runtime (d3drm.dll) is bundled with the release and deployed alongside the binaries. System DirectDraw headers come from the Windows SDK. No external DirectX SDK download needed.
Why x86 only?
The original code casts pointers to long throughout. On 64-bit Windows, long is 32 bits but pointers are 64 bits — this would truncate pointers and crash. Fixing this would require extensive changes across the entire codebase. The platform runs as a 32-bit process on 64-bit Windows via WoW64, which works perfectly fine.
What is Direct3D Retained Mode?
D3DRM was a high-level 3D API that Microsoft shipped in the late 1990s. It provided a scene graph, mesh loading (.x files), hierarchical transforms, and texture management. Microsoft deprecated it after DirectX 6 — you know, around the time they decided everyone should just write their own scene graph from scratch using Immediate Mode. The runtime DLL still works on modern Windows when loaded app-locally, which is either a testament to Microsoft's backwards compatibility or a sign that nobody has thought to remove it.
What TCP port does VWorlds use?
VWorlds uses registered port 2525. HTTP tunneling is not supported — port 2525 must be open for client-server communication. This was true in 2000 and remains true today. Some things never change.
What is the Spuck bone animation system?
An unfinished skeletal animation system from the original 1999 code. The COM infrastructure (CJoint, CCharacter, CPuppet) was there but the callback function pfUpdateJoint was NULL. It was built for articulated characters — skeletal animation in a virtual world, years before it became standard. But the team ran out of time and left the plumbing unconnected.

The revival project wired it up and created the Actor exemplar with InitializeActor, SetJointRotation, and PlayAnimation methods. Twenty-six years after someone wrote the joint system and left pfUpdateJoint as a NULL pointer, bones finally move in VWorlds.
What about the original HTML client?
The complete original 1999 HTML+VBScript+ActiveX client UI exists at Local Content\Client\Basic\. It's a frameset with Render, Chat, and Controls panels, plus 40+ VBScript files. The revival's webclient.exe hosts it in a WebBrowser control (IE/Trident) and it actually runs — login, rendering, Things tab, sound all functional. Turn off hardware acceleration in Options for rendering to work.
What is the license?
The Microsoft Research Source License (MSRSL). You may do anything you want with the software for non-commercial research or teaching purposes, free of charge. Modifications must be shared under the same terms. Modernization patches are provided under the same license.
Will you allow commercial use?
The MSRSL does not allow commercial use of any portion of the SDK. This was the original team's decision and the revival project honors it. If you'd like to license the platform for commercial use, you'd need to ask Microsoft Research — though given that the group disbanded over two decades ago and the technology is from the Clinton administration, it's probably not their top priority.
How do I report bugs or contribute?
File issues or contribute via GitHub. The original process involved emailing a Code Submission Agreement to code@vworlds.org and posting to the platform-dev mailing list. We've modernized the contribution process along with the code.
I'm running WinME or IE5.5 and having graphics problems. What's wrong?
[From the original FAQ, preserved for historical amusement]

There was a problem with the rendering control not properly un-hiding after logon on some machines. The original fix was to replace 3dview.htm with a patched version from vworlds.org/files/3dviewpatch.zip.

If you are still running Windows ME in 2026, we salute you. You may also want to check that your Y2K patches are up to date.
How can I submit crypto related code?
[From the original FAQ]

The original code submission agreement prohibited crypto code because the license didn't have export restrictions. This was 2000 — a time when the US government classified strong encryption as a munition and you needed a license to export PGP. The world has moved on. Mostly.
What's with all the downloads?
[From the original FAQ]

The original SDK was split into several separate downloads due to third-party licensing requirements. The revival project bundles everything into a single release zip. We have solved the download problem that plagued the year 2000. You're welcome.
