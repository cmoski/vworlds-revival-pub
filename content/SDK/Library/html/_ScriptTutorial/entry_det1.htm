Step 1 of 4
Step One: Objects and Containers

When an object has another one inside it, it is said to be a container of the inner object. For example, the room object is the container of all the objects inside it, which in turn may contain other objects (as is the case for many of the lessons).

Objects can receive events from the world telling them that an object has changed its container. There are events that Virtual Worlds sends to every object that mark when an object leaves a container and when it enters a container, but usually objects do not have the event-handling methods added to them to make use of the events.
a For this lesson we go into a different room in the tutorial. You should see a door appear in the 3-D view.
a Right-click that door and select the menu command Go Through on it. You should see a spinning cube and be transported into a new room.
$ Notice that the Object Explorer list has changed to indicate the content hierarchy of this new room. There should also be another portal visible. This is a portal back to the main lesson room.
a Go through the portal and watch the Command Window to see if the scripts on these portals detect that your avatar passes out of the sub-room container and into the main lesson room container, and then go on to the next step.

Step 3 of 4
Step Three: Generic and Specialized Methods

In the next step, we take a look at the custom methods do_surprise and end_surprise that have been added to the two portals used in the example.
$ You see four distinct subroutines in the code.
a Examine each of the portals of both rooms in the Object Explorer.
$ Both have methods called do_surprise and end_surprise that come from the same module, but notice that their script subroutines for these methods are different.
$ This is to show you that you can have one method, such as OnPeerEnter, that contains script common to many objects (the portals run the same script at this point), and that this method can, in turn, call a more specialized behavior specific to a need.
$ If we wanted the portals to behave wildly differently from each other, we could place entirely different code in their respective subroutines for the do_surprise and end_surprise methods.
$ If we had a hundred objects, we could make 90 of them use the same scripts for their responses to a particular event, give 10 of them totally distinct scripts, and only need to write about 22 subroutines instead of 400.
~ All we would have to ensure is that each portal had some kind of do_surprise and end_surprise methods, even if only a few were made specialized by pointing them at individualized subroutines.

Step Four: Detecting Taking and Dropping

The scripts we examined before used a conditional statement to make sure that they ran the custom "surprise" methods only when the object that changed was an avatar.
a To verify this, navigate your avatar through the Lesson_7_portal into the sub-room.
a You should find an object called "key" listed in the room's Contents Tab. (You can find that under the tab marked with a house, in the top right of the Virtual Worlds window.)
a While watching the Command Window, click on the word "key" in the list and select Take from the sub-list of options that expands below it. You have now moved the object out of its container (the room) and into a new container (you!).
$ The portal script detects that something has left its container, but it has no particular behavior associated with that action. You could use the same mechanism to detect when objects are picked up or dropped in your world, so that you can make special things happen in response.
a Try dropping the key object from your contents, and picking it up, to see the code at work. With the key in your contents, switch back to the main room and Drop it there. The same code functions just as well in this room, making it possible to track a special object anywhere in a world.
a Now take a look at the scripts that will run if the object that is being sent to the scripts passed the test that checks whether it is an avatar.
$ They do not contain much special code at the moment, but they do move the avatar to a set point and trace into the Command Window.
a Try to think about how you might customize them further, to do things like invoking methods on other objects so that they interact with the avatar.
