Step 1 of 4
Step One: Script Modules

The Object Explorer provides a comprehensive view of all objects in a world. We can start looking at object scripting by examining the modules that contain scripts. In the Object Explorer, notice a long vertical listing of items in the left side of the window. The list describes the object hierarchy of this tutorial world. Each item in the list is a Virtual Worlds object that this tutorial uses. Script modules, listed close to the top under the heading Modules, are accompanied by a globe icon. These objects have no editable features in Virtual Worlds; they are markers for the text files that contain the script code. By associating subroutines inside these files with objects in the virtual world, you can give these objects interactive behavior. These script files are located in this directory (assuming you chose the defaults during installation): C:\Program Files\Microsoft Virtual Worlds\Local Content\SDK\Library\vbmodules. A module contains information about how to get to the script file. All Virtual Worlds files must be referenced by valid URL paths for a Virtual Worlds client and server to access them.

In most cases, we do not recommend that you place scripts for clients on a local machine path, because if you want to make changes, you must alter the file on every client. However, if you place your scripts on a web server, you need to change only that copy, and the changes are distributed to all clients when you right-click the module to reload it.
a Right-click anywhere in the Modules list in the Object Explorer. Choose New Module.
a A pop-up web page will appear for you to enter information about a new module. For the name, enter Lesson_1. For the Module URL, enter the line below exactly (note that the line may appear broken, but is one path):
file://C:\Program Files\Microsoft Virtual Worlds\Local Content\SDK\Library\vbmodules\lesson_1_adding_methods.vbs
$ You can copy and paste this path line by highlighting it with the cursor, then right-clicking the selection, choosing Copy, then right-clicking in the Enter Module URL text area, and then choosing Paste.
a Leave the other "execute" option unchanged and click OK. Your new module Lesson_1 should appear in the Object Explorer now.
a Open the vbmodules folder located in \Local Content\SDK\Library\ and read the files with Notepad or another text editor. (Their contents are included in the HTML of these lessons for easier reference.) ~Be careful not to alter them!
a Look at lesson_1_adding_methods.vbs to see that it is, after all, just a text file with a script subroutine in it.
a Go on to the next step.

Step Two: Object Hierarchy

The Object Explorer's vertical list also lists the room you are logged on to: it is called Tutorial Main Room, and it appears in the list next to a blue cube. Below this, you can see the objects contained in the room you are logged on to; they appear in the list below the room's name, usually accompanied by small red pyramids. These items are actual graphical objects users can see and interact with inside the world. Ignore the door and the people symbols for now; they are portals to other rooms and your own avatar, respectively.
a Try selecting different Virtual Worlds objects listed below the room by clicking them.
$ The properties and methods of the selected object appear in the "properties" and "methods" windows in the Object Explorer.
a Scroll through these lists to see the kinds of properties and methods the selected object possesses, but do not try to edit any of them yet.
$ You may notice that some tutorial objects are further expandable by hitting the + button next to them. The contents of these expandable objects appear below them in the hierarchy, and are also Virtual Worlds objects.

One object may contain many helper graphical objects within it that make complex behaviors possible; you will see examples of such compound objects in many of the lessons. Some objects contain other objects, which, in turn, contain still other objects, and so on. One benefit of nesting objects inside each other is that changing the position of the "top" or "owner" object from one location to another in a Virtual World causes all the objects inside it to move to the owner's location, yet keep their spatial relationship to each other. Each top-level object in the room usually contains all the graphical objects for the lesson inside it (with one exception), or, if only one graphical object is needed, just that top level object will suffice.

When you select a lesson in this web page, the room uses custom methods from the script module Lesson_Manager to take care of positioning the objects for each lesson by moving the current lesson's top level object into your view, and then putting it back out of view when you have moved on to another lesson. All the things listed inside that object move with it, which makes the tutorial "housekeeping" very easy. (You don't need to worry about these methods, as they are purely for logistical purposes, but the curious will find the file lesson_manager.vbs in the vbmodules folder inside the sdk library.)

aTake a minute to look at the object hierarchies for the tutorial room objects, and then go to the next step.

Step 4 of 4
Step Four: Managing Methods

Most of the methods in an object's methods list show a module prefaced by the word <COM>. These are system-level, hard-wired methods that you are unlikely to modify to make custom behaviors, although you might call them within your custom scripts to make use of their actions.

$Look closer for the methods listed that have modules that do not start with <COM>. Usually, they start with the word "lesson" in the files associated with this tutorial. These are custom methods that operate by scripts written for this tutorial. (Such methods are sometimes called instance methods.)

~It is very important to remember that because OnLButtonUp is a generalized method that handles a standard Virtual Worlds event, its name can not be customized. You can, however, associate it with a subroutine of any name.

Pre-defined methods that handle events are often described as built-in methods, yet there is still a lot of work to be done if you want to make use of them. In fact, if you look at many of the other objects in the Object Explorer, you will see they do not have the built-in LButtonUp method listed. (Neither did the planets until you added it to them in the previous step.) This is because most event-handling methods do not come defined with any coded behavior; because there are far too many of these available to each object to make this practical.

A script subroutine within an external script file (such as sub planet_click inside lesson_1_adding_methods.vbs inside the sdk/library/vbmodules folder) must be associated with the method manually using the Object Explorer. Similarly, a predefined method, such as you might find in the Programmer's Guide, must also be typed into the Object Explorer by hand.

Each new method you add in the Object Explorer has a module and a sub, which describe to Virtual Worlds the way this method finds a script that it has been programmed to run. The module shows Virtual Worlds where to look for the file, and the sub shows it which subroutine in the file to use. Remember that when you add custom methods to an object, Virtual Worlds always requires a script to tell the system exactly what you want the object to do; Virtual Worlds does not add the method without such a script.

World builders and scripters will need easy access to the actual text files (such as are found in the vbmodules folder) to know the names of the subroutines in them, and associate them with a built-in Virtual Worlds method. Virtual Worlds will not retrieve the subroutine names for you, however. If the method you are adding has the same name as one of the built-in Virtual Worlds methods, such as are documented in the Programmer's Guide, this association is called overriding the method. A method does not have to override a built-in feature; you can add a method called "dancing-fool" or whatever you like to make a custom behavior on an object and associate it with a script. To allow users the ability to use that method, though, you will usually override one of the built-in methods discussed in the User Interactions section of this tutorial. That method's name would then call your custom method to action.

One text file (and one Virtual Worlds module) can hold as many script subroutines for as many different objects as you like. In fact, one file/module could have contained all the scripts for every lesson in this tutorial. That would have been very difficult to organize conceptually for the world builders, however, since the file is not required to describe the function of each subroutine (unless it is commented well), and it can be confusing to read through the file, which is generally script code.

aTake a closer look at the relationship between methods, VBScript modules (not <COM>) and the subs they use by clicking them in the Object Explorer, and then you will be ready to start on the rest of the lessons.
