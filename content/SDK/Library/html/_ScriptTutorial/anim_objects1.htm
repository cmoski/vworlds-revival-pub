Step One: Compound Objects

This set of objects introduces the Virtual Worlds timer capability. We use clickable object behavior in conjunction with a timer to make a self-timing set of objects that a user can start and stop with mouse clicks.
a In the Virtual Worlds window you should see a large 3-D timer object with panels that read Start and Stop. These panels are clickable and their scripts activate the center arrow object. Click the left-hand panel to activate the timer.
a When it has finished rotating and changing color, click it again.
$ Notice that this object knew how to turn itself off when it completed its task! This time, when you click Start, try to click the Stop panel before the timer goes around and notice that it has a different behavior. Try different combinations of start and stop, and watch the messages in the Command window as the scripts run.
$ If you expand the object marked Lesson_3 in the Object Explorer, you can see that it contains three objects-- the Start and Stop panels, and the arrow. The Lesson_3 object itself is the ring shape that combines and connects all of these parts. These are all Virtual Worlds objects in their own right, but they work together in scripts to make a virtual machine that has moving and animating parts.

Although the timer appears as one graphical form, it is an illusion of the design. You can move the pieces away from each other with the Geometry Editor, and they should function precisely as they did before. (If you attempt this, please try to move them back together for the next reader!) Different component objects within the whole composition have different custom methods attached to them that create the corresponding part of the composition's functionality. Some are clickable and manipulable, in the same way links on a web page have separate tags even though they appear as part of one unified document.
a You can go on to Step 2 to examine the relationship of the component objects and scripts to the whole sequence.

The Virtual Worlds object hierarchy allows objects to be contained within other objects even when their graphics look distinct. In this case, the center ring object owns the arrow, start and stop objects, but the methods run on the 3 components, not the owner.
$ Take another look at this object's ownership tree, using the Object Explorer. In the next step, we will look at the subroutines for the start and stop panel objects you can see below the Lesson_3 object in the tree.

The start_timer routine below is used by a "child" object (start_button) of the Lesson_3 object to make a timer. Step 4 of 4
Step Four: Timers, Objects, and Persistence
~ Note that the timer is not really a part of any object. It is a Virtual World's component that can be started anywhere, even, as we saw in this lesson, by a completely different object than the one that has the method the timer will be calling! You can run as many timers calling any methods on any objects as you need. A timer provides the primary means of doing a repeatable task on an object that occurs through time, and managing the stopping and starting of the timer in the method that it is calling is very important to prevent timers that can run on too long.
  A timer is -not- a persistent part of the world unless it is being run on the server, and the server has active clients connected to it (we will discuss server side methods that use timers in future lessons). These timers run only on the client, that is, they run on a user's local machine to provide behaviors for the objects. If an object has a timer running on it and a user logs off the world, that timer is destroyed automatically. When they log back on, the object will not still have a timer running on it. However, it is possible to create special methods that can detect a user logon and reactivate timers to give the illusion of persistence, using different Virtual Worlds event-handling methods.
