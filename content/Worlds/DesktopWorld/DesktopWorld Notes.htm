DesktopWorld Notes

 

The DesktopWorld sample application implements some interesting variations on standard Microsoft Virtual Worlds objects and some fairly complex interactions between the world and the DHTML client.

 

This document describes some of these unique features and provides insight into their implementation.

 

Multipart Avatars

 

Avatar objects themselves are not visible in DesktopWorld. Each avatar carries around a collection of five visible bodyPart objects: shoes, pants, shirt, skin and hair. Six body types are available, each a collection of these five parts. The individual parts can be independently colorized, but they are not interchangeable; for example, the pants of the Sean body cannot be attached to the shirt of the Russ body. This is only because these avatars were intended to look somewhat realistic. For a world with more fanciful avatars, the code could be modified to allow users to assemble their own avatars from interchangeable bodyParts.

 

Since an avatar's bodyParts are in its contents, navigating them through the world is not an issue; the parts simply move with the avatar. But to get the avatar to perform a gesture the five parts must all perform the gesture in unison. Users do not actually click on avatars, they click on bodyparts and the bodypart click events are passed from the bodypart to the avatar.

 

Automated Navigation

 

A system of paths and boundaries allows avatars to be moved automatically (guided) from one area of DesktopWorld to another. The bounded areas are the Resource Desk, Seattle Desk, Discussion Area, Game Room, Music Garden, Gift Carts area, individual Flower Cart and Balloon Cart, and the central area of the Lobby that is not within the other areas.

 

Each area is surrounded by a passable boundary. Boundary collisions are detected when an avatar passes from one area to another. When the user selects auto-navigation in the UI to move to a specific area, the avatar is moved along a predetermined path between its current area and the new area. This is handled in DesktopWorld by statically storing all possible paths between all areas.

 

The data structures describing each area used for navigation contain some information about each area, are used to update a status line in the UI and to display web pages pertaining to the different areas.

 

Animation

 

The methods for navigating avatars on predetermined paths were originally written for an earlier version of Virtual Worlds. Although it still works, using the built-in animation methods of Virtual Worlds version 1.5 would make it much simpler and more efficient.

 

Other objects animated with pre-1.5 techniques include the user avatar's camera and the clickable objects in the Music Garden. The camera movement path is calculated in real-time based on ellipse formulas. The Music Garden objects iterate through lists of position and direction values. The Music Garden objects have small, repetitive animations that they perform while idle, and others that they do when clicked.

 

Client-to-Client Communication

 

Communicating between two users via object methods, and giving feedback on both of their clients, can be somewhat confusing because it involves a sequence of client and server methods. User 1 runs a client-side method (so it can fire a UI event on client 1 for feedback), which runs a server-side method, whose only purpose is to run a second client-side method, which runs on all clients but only takes action if its user is user 2. This general technique is used to implement the Make Friends and Ignore features.

 

Interaction Between Client Pages

 

The DesktopWorld client consists of an outer page (client.htm) with some iframes containing several other pages, many of which respond to events taking place in the world. Several mechanisms are in place for these pages to communicate with each other and the world.

 

One handles UI events fired by in-world objects. Each page that needs to receive UI events must have a sub called HandleUIEvent, and must add its window to an array of window pointers defined in Client.htm. The Virtual Worlds client object, also defined in client.htm, has an OnUIEvent handler that runs when the client object receives UI events from the world. This method iterates through the list of window pointers and calls each window's HandleUIEvent sub.

 

Another communications mechanism uses global variables in Client.htm and buttons in ClientMainToolsIFrame.htm to display information in the browser frame (the area below the toolbar buttons). For example, clicking the People button opens a page called people.htm that displays a list of avatars, then checks a global called CurrentAvatarId and displays that avatar's profile. Other pages in the client can trigger the page to display a specific avatar's profile by setting the global variable and calling the People button's Click method. ChatHistory.htm does this when you click on an avatar thumbnail image in the chat history. Client.htm does it to handle the "ShowProfile" UI event, which is fired when you pick "Profile" from an avatar's right-click menu in the 3d renderer.

 

Object Ownership -- Gifts and Notes

 

Enabling avatars to exchange dynamically created objects presents the problem of eventually getting rid of these objects. Virtual Worlds security allows only the user that created the object to destroy it. To enable users to delete notes and gifts from others, these objects are created by the recipient. Instead of the sender avatar creating an object and sending it, the sender merely passes information to a method of the recipient avatar, which creates the actual gift or note. The recipient owns the object and is able to destroy it later.

 

Holding Gifts

 

Avatars in DesktopWorld are can display one gift at a time from their list of gifts. This is referred to as "holding" the gift. In addition to the 2d graphics displayed in the ShowGift.htm page, each gift has a single representative graphic that simply shows the type of gift (flowers or balloons). Setting a gift's IsVisible property = true makes this graphic appear in front of the user. All gifts are basically in the same position within the avatar; they are only made visible one at a time.

 

Timing Issues

 

As with all client/server apps, programming for Virtual Worlds requires an awareness of what is happening on the client and on the server, and when. Errors will occur if scripts in web pages try to interact with in-world objects that have not been marshaled over to the client yet. A number of procedures in client page scripts use the window.setTimeout method to re-execute themselves until the objects they need are available. For example, in various client pages the window_onLoad sub calls itself repeatedly until a global variable called IsWorldLoaded is set TRUE by Client.htm. This ensures that the pages get initialized properly by making them wait for the data to be available.

 

Several helpful tools for making sure world objects are ready are to check value of the IsObject and IsNull functions, the IsStub method of the Virtual Worlds Thing type, and to compare an object variable to Nothing. If you get an error message like, "Object does not support this property or method," a timing problem is a likely suspect. Think about using window.SetTimeout in web script code to reattempt a procedure until data is available, or firing a UI event within a client-side method of an in-world object to signal the UI when it is safe to run the code.

 

Client References to In-World Objects

 

Much of the DesktopWorld client's functionality involves handling lists of in-world objects and associating them with pieces HTML code. This is generally done by using an object's id property, which is an integer value, to create an ID value for an HTML tag. Every object in Virtual Worlds, including avatars, has an id property. Using an object id to get an actual object from Virtual Worlds involves using an undocumented world method: world.objectById(id), which returns a pointer to an object. The problem with this method is that it generates an error if there is no such object id, which happens if the object has been deleted. Client.htm provides a function GetObjectById which uses "on error resume next" to safely call world.ObjectById, returning a value of Nothing if no object is found.

 

Clicking Through Avatars

 

One problem with implementing avatar gestures is that the avatar graphics must be wide enough to display the largest gesture. Because of this there is a transparent area around the avatar which the renderer considers part of the avatar even when the avatar is not gesturing. When you try to click on an object that lies behind this clear area, you are actually still clicking on the avatar. DesktopWorld uses a special technique to enable the user to click on objects behind the avatar.

 

When the user clicks on an avatar body part, the bodypart_OnLButtonUp method (in avatar_module.vbs) fires a UI event called "avatarLButtonUp". The HandleUIEvent procedure in ClientRendererIFrame calls renderroot.vwrender3d.hittest, an undocumented method of the 3d renderer object. This method returns a property list of all graphics frames that lie behind the mouse pointer. The HandleUIEvent procedure iterates through this list until it finds an object that is neither an avatar nor contained in an avatar. If it finds such an object, it simply calls that object's OnLButtonUp method, simulating a click on that object.

 

Note that this click-through behavior applies to clicks on the visible body parts of the avatar as well as the transparent areas. The renderer simply does not distinguish between the two.

 

Some HTML Editors May Induce "Unsafe Control" Warnings

 

Some controls used in the DesktopWorld client are not marked safe for scripting -- for example, the 3D renderer control in ClientRendererIFrame.htm. Since these controls are not scripted in the client they should generate no errors or warnings as used. However, some HTML editors may helpfully add parameters to controls, in effect scripting the controls. This will generate safety warnings when the client is started. If you edit the client HTML files, or even examine them by opening them with an editor, use an editor such as Microsoft Visual Interdev or Notepad.
